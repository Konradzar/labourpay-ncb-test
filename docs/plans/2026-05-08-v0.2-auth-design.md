# Design — V0.2 NCB Better Auth integration

**Date**: 2026-05-08
**Author**: Konrad (collaborated with Claude)
**Status**: Approved — ready for implementation plan
**Builds on**: V0.1 design + plan (paused) + V0.1.5 (shipped on commit `4d342d1`)

---

## Purpose

Wire NCB's Better Auth into the labourpay-ncb-test app so:

1. The app requires login before any worker data is visible.
2. V0.1's `updateWorker` and `deleteWorker` Server Actions, currently 403'ing because NCB requires session cookies for PUT/DELETE on `/api/data`, become functional.
3. The codebase advances toward V0.3 deploy-readiness (RLS flip is V0.3, but session forwarding lands here).

This is V0.2 = "Real per-user login via NCB Better Auth" per the V0 design doc's deferred-features table.

## Decisions log

| # | Decision | Reasoning |
|---|---|---|
| 1 | Single admin scope (Konrad only) | V0 framed as a learning instrument, not a multi-tenant SaaS. Smallest viable. Q1=(a). |
| 2 | Login + forgot + reset pages, no signup UI | NCB requires reset for email auth (mandatory). Konrad's account is created via one-off curl to NCB's `/sign-up/email` endpoint. Q2=(c). |
| 3 | Anonymous → /login redirect (gated app) | Konrad is the only intended user; no anonymous audience. Simplest mental model. Q3=(b). |
| 4 | V0.1 (edit + delete) unblocks as part of V0.2 | Adds ~30 min, lands paused work. PATCH→PUT bug + session-cookie forwarding fix together. Q4=(a). |
| 5 | Defer RLS flip to V0.3 | Workers stays `public_readwrite`; new workers continue with `user_id = NULL`. Backfill + flip happens at deploy time. Q5=(b). |
| 6 | Architecture: route groups + layout-level guard | Cleanest separation, scales to V1, idiomatic Next.js. Approach (A). |
| 7 | Auth proxy is one catch-all `app/api/auth/[...path]/route.ts` | NCB's user-auth API has many endpoints; one file handles all. Per integration guide template. |
| 8 | Session check: `(app)/layout.tsx` calls NCB `/get-session` server-side per render | Matches integration guide. Real validation (not just cookie-presence). Single-admin scale: cost is negligible. |
| 9 | `ncbAuthFetch` adds session-cookie forwarding in addition to existing Bearer auth | Strictly additive — adds permissions, removes none. Backward-compatible with all existing anonymous read paths. |
| 10 | Email delivery for password reset relies on NCB's built-in service | We don't configure SMTP. If NCB's delivery fails for the user's domain, the flow is operationally broken even though structurally complete. |

## Auth flows

### Login

1. Anonymous user hits any gated route → `(app)/layout.tsx` runs server-side
2. Reads cookies via `headers()`, calls `getSessionUser(cookies)`
3. Returns `null` (no session) → `redirect("/login")`
4. `/login` (`(auth)/login/page.tsx`) renders; user submits email + password
5. Form POSTs to `/api/auth/sign-in/email`
6. Catch-all proxy forwards to `${NCB_AUTH_API_URL}/sign-in/email` with body
7. NCB validates, returns `Set-Cookie` (with `__Secure-` prefix) + body with user info
8. Proxy strips `__Secure-`, strips `Domain`, removes `Secure` flag, sets `SameSite=Lax`, forwards to browser
9. Browser stores cookies → `router.push("/")` → `(app)/layout.tsx` runs again, this time with cookies → `getSessionUser` returns user → page renders

### Sign-out

1. User clicks "Sign out" in `(app)/layout.tsx` header (a `<form>` with Server Action)
2. Server Action calls `/api/auth/sign-out` (proxy → NCB)
3. Clears local cookies via `cookies().delete(...)` regardless of NCB response
4. `redirect("/login")`

### Password reset (two stages)

**Stage A — request:**
- User clicks "Forgot password?" on `/login` → `/forgot-password`
- Form (Client Component because it needs `window.location.origin`) POSTs to `/api/auth/request-password-reset` with `{ email, redirectTo: ${origin}/reset-password }`
- Proxy forwards to NCB; NCB sends email
- Page shows "Check your email if the address is registered"

**Stage B — submit:**
- User clicks email link → `/reset-password?token=X&db=Y`
- Page reads `token` + `db` from query params; user enters new password
- Form POSTs to `/api/auth/reset-password` with `{ token, newPassword }`
- On success: redirect to `/login`

## Session-cookie forwarding mechanism (the V0.1 unblock)

Modified `lib/ncb-utils.ts` `ncbAuthFetch`:

```typescript
import { headers } from "next/headers";

export async function ncbAuthFetch(pathSuffix: string, init?: RequestInit) {
  const cookieHeader = (await headers()).get("cookie") ?? "";
  const sessionCookies = extractAuthCookies(cookieHeader);
  return fetch(`${dataApiUrl}${pathSuffix}`, {
    ...init,
    headers: {
      "Authorization": `Bearer ${secretKey}`,
      "Content-Type": "application/json",
      "X-Database-Instance": instance,
      ...(sessionCookies && { "Cookie": sessionCookies }),
      ...(init?.headers ?? {}),
    },
  });
}
```

**Backward compat:** when no session cookies present, behavior is identical to V0 (Bearer-only). All existing anonymous read paths keep working.

**V0.1 unblock effect:** `updateWorker` and `deleteWorker` Server Actions run in the request context where session cookies exist (because user is logged in to reach the page that triggered the action). Cookies flow through automatically.

## Error handling

| Scenario | UX |
|---|---|
| A. Wrong email/password on `/login` | Inline red text under form; password cleared |
| B. Unknown email on `/forgot-password` | Always show "Check your email if registered" (no leak) |
| C. Expired/invalid token on `/reset-password` | Inline: "Link expired. [Request a new one]" |
| D. Network error on any auth call | Inline: "Couldn't reach the server. Try again." |
| E. `getSessionUser` throws in layout | Default Next.js error boundary; user retries |
| F. Session expires mid-use | Action throws → boundary; user refreshes → layout catches missing session → redirect to /login |
| G. Already logged in, visits `/login` | Just shows form (V0.2 minimum; could redirect to `/` later) |
| H. Sign-out NCB call fails | Cookies cleared anyway, redirect to /login |
| I. Reset email never delivered | "If you don't receive within 5 minutes, contact support" — for V0.2 single-admin, that's you |

### Conventions

- Field-level errors inline, `#d63031`, `0.85rem`, same as AddWorkerForm
- Form-level errors as banner above submit button
- All NCB error bodies surfaced raw to dev console for debuggability
- No custom `error.tsx` boundaries (V0.3 if needed)

### Edge cases knowingly NOT handled

- CSRF (Next.js Server Actions handle automatically)
- Login rate-limiting (rely on NCB)
- Account lockout
- Email enumeration timing attacks
- Multi-device session management
- Session refresh / "stay signed in"

## Forward compatibility

V0.2 prepares us well for:
- **V0.3 deploy:** session forwarding already in place; only RLS flip + backfill + (optional) deploy-specific config remains
- **V1.0 multi-table:** `(app)/layout.tsx`'s session check applies to any future page added inside the group; auth surface scales linearly

V0.2 doesn't prepare us for:
- Real multi-tenant signup (would need to remove "single admin" UI assumption + add open signup form)
- OAuth providers (Google) — would need additional `app/auth/callback/page.tsx` + provider-specific UI gating
- Email OTP — different auth flow, separate UI
