# Design — V0.3 Deploy to Greta + RLS Flip + Call-Path Migration

**Date**: 2026-05-08
**Author**: Konrad (collaborated with Claude)
**Status**: Approved — ready for implementation plan
**Builds on**: V0.2 design + plan (shipped on commit `6ab28a9`)

---

## Purpose

Take the localhost-only V0.2 app and ship it to a public URL on Greta (Netlify-backed) with proper authentication enforcement at every layer:

1. **Database layer** — workers table flips from `public_readwrite,shared_readwrite` to `private`, so NCB itself filters rows by `user_id`.
2. **Application layer** — all NCB I/O moves server-side (Server Components / Server Actions / `ncbAuthFetch`); the browser stops talking to NCB directly.
3. **Route handler layer** — supporting upload routes (`/api/upload-url`, `/api/perceptpixel-upload`) gain session checks so anonymous callers can't burn S3 / PerceptPixel credits on a public URL.

This is V0.3 = "Greta hosting" per the V0 design doc's deferred-features table, expanded to cover the security migration that must accompany any public deploy.

## Pre-state and target state

### Currently (commit `6ab28a9`, after V0.2)

- **RLS:** `workers` = `public_readwrite,shared_readwrite` (verified via NCB MCP `execute_sql` on `ncba_rls_config`)
- **Data:** 9 workers, all `user_id = NULL` (verified at end of V0.2: deleted ids 5 + 7, remaining 9)
- **Auth:** `(app)/layout.tsx` enforces session at the page-render layer; unauth → `/login`
- **Server-side helpers:** `ncbAuthFetch` already forwards Bearer + session cookies (V0.2)
- **Anonymous call sites still in production code:**
  - `app/(app)/page.tsx:30` — direct `fetch(${dataApiUrl}/read/workers...)` with no auth headers
  - `app/(app)/workers/new/AddWorkerForm.tsx:185` — browser POSTs to `/api/public-data/create/workers`
  - `app/api/upload-url/route.ts` — anonymous mints S3 presigned URLs
  - `app/api/perceptpixel-upload/route.ts` — anonymous proxies to PerceptPixel
- **Hosting:** localhost:3000 only

### After V0.3

- **RLS:** `workers` = `private`
- **Data:** 9 workers backfilled to bootstrap user's UUID; new workers carry `user_id` automatically (NCB sets it from the session)
- **Server-side everywhere:** browser never carries an NCB credential; all NCB I/O is `ncbAuthFetch` from Server Components/Actions
- **Auth-gated upload routes:** session-check returns 401 to anonymous callers; logged-in callers behave exactly as before
- **`/api/public-data/[...path]/route.ts` deleted** along with its dependent helpers (`proxyToNCBPublic`, `getRlsPolicies`, `extractTableFromPath`, `allowsPublicRead`, `allowsPublicWrite`, `requiresOwnerScope`) if they have no other callers
- **Hosting:** live on a Greta-Netlify subdomain, env vars set in Netlify, CORS allows production origin on both NCB and S3

## Architecture diff

```
                  V0.2 (now)                          V0.3 (target)
─────────────────────────────────────────────  ──────────────────────────────────────
Browser → POST /api/public-data/create/workers Browser → <form action={createWorker}>
       → public-data proxy                            → Server Action (RSC wire format)
       → NCB anonymous (public_readwrite)             → ncbAuthFetch (Bearer + cookie)
                                                      → NCB authenticated (private RLS)

Server Component (list page) → fetch(NCB url)  Server Component → ncbAuthFetch
                  (anonymous, public_readwrite)                   (authenticated, private RLS)

POST /api/upload-url (anonymous)               POST /api/upload-url
   → mint S3 presigned PUT                        → getSessionUser; 401 if absent
                                                   → mint S3 presigned PUT

POST /api/perceptpixel-upload (anonymous)      POST /api/perceptpixel-upload
   → forward multipart to PerceptPixel            → getSessionUser; 401 if absent
                                                   → forward multipart to PerceptPixel
```

The auth proxy `app/api/auth/[...path]/route.ts`, the `(app)/layout.tsx` session guard, and the login/forgot/reset pages are unchanged from V0.2. No new abstractions; this is a migration milestone, not a feature milestone.

## Order of operations

The wrong order breaks either localhost or production. Do this in sequence:

1. **Code migration** (locally, no DB / deploy changes yet)
   1. List page: swap `fetch` → `ncbAuthFetch`
   2. New Server Action `createWorker`
   3. Refactor `AddWorkerForm.handleSubmit` to call `createWorker` instead of `/api/public-data/create/workers`
   4. Add session guards to `/api/upload-url` and `/api/perceptpixel-upload`
   5. Delete `app/api/public-data/[...path]/route.ts`
   6. Verify dead helpers in `lib/ncb-utils.ts` and delete if unused
   7. `npx tsc --noEmit` clean; manual smoke on localhost (still works because RLS is still permissive — proves the code change alone didn't break anything)
2. **Backfill `user_id`** via NCB MCP `execute_sql`:
   - First: `SELECT id FROM ncba_user WHERE email = 'konradzar@gmail.com'` — capture UUID
   - Then: `UPDATE workers SET user_id = '<that-uuid>' WHERE user_id IS NULL` — should affect 9 rows
   - Verify: `SELECT id, name, user_id FROM workers ORDER BY id` — all 9 show the same UUID
3. **Flip RLS** via NCB MCP `set_rls_policy`:
   - `set_rls_policy(database='35120_labourpay_ncb_test', table='workers', policy='private')`
   - Verify: `SELECT table_name, policy FROM ncba_rls_config WHERE table_name='workers'` shows `private`
4. **Local smoke test (full V0.3 happy path on localhost)**:
   - Open incognito → `/` → redirect to `/login`
   - Sign in as bootstrap user → list shows all 9 workers
   - Click a worker → detail page loads
   - `/workers/new` → fill + submit → redirects to list, new worker visible
   - Edit (when wired) / delete a worker → behaves as in V0.2
   - Sign out → redirected to `/login`
5. **Update `MANUAL_TEST.md`** with V0.3 scenarios 23–28
6. **Commit + push to GitHub**
7. **Konrad-driven (UI work I can't do)**:
   1. Connect Greta to the GitHub repo
   2. Set env vars in Netlify dashboard (checklist below)
   3. Trigger first build; iterate on any build errors
   4. Once Netlify URL is known, configure CORS:
      - NCB: add origin via NCB MCP `set_config_value` if available, else NCB dashboard
      - AWS S3: add origin to bucket CORS `AllowedOrigins` (existing config in S3 console)
8. **Production smoke test** — re-run step 4 against the Greta-Netlify URL
9. **Update `README.md`** to record the live URL and V0.3 ship status

Steps 1–6 + 8 + 9 are mine. Step 7 is yours, with me on standby for build-error triage.

## CORS checklist

### NCB allowed origins

V0.2's auth proxy forwards `Origin` and Better Auth's CSRF check rejects unknown origins. Add the production hostname (e.g. `https://<sitename>.netlify.app`) to NCB's allowed origins for the instance. Localhost stays whitelisted for dev.

### AWS S3 bucket CORS

Existing CORS config has `AllowedOrigins: ["http://localhost:3000"]` (or similar). Add the production origin so browser-direct PUTs from the presigned URL work. Method allowlist already includes `PUT`.

```json
{
  "CORSRules": [
    {
      "AllowedHeaders": ["*"],
      "AllowedMethods": ["GET", "PUT"],
      "AllowedOrigins": [
        "http://localhost:3000",
        "https://<greta-netlify-subdomain>.netlify.app"
      ],
      "ExposeHeaders": [],
      "MaxAgeSeconds": 3000
    }
  ]
}
```

### PerceptPixel

PerceptPixel uploads happen server-side (`/api/perceptpixel-upload` proxies); no browser-direct call, no CORS concern.

## Risks & rollback

| Risk | Likelihood | Mitigation / Rollback |
|---|---|---|
| Greta/Netlify build fails on Next.js 16 | Medium | Netlify's Next.js Runtime supports App Router + Server Actions. If it fails, pin `engines.node` in `package.json` to `>=20`. Final fallback: `next start` on a $5/mo VPS — 30 min of work. |
| RLS flipped before backfill — workers vanish | Low (strict ordering) | Re-run backfill SQL: `UPDATE workers SET user_id='<uuid>' WHERE user_id IS NULL`. If panicked, temporarily flip RLS back to `shared_readwrite` while diagnosing. |
| Browser uploads blocked by S3 CORS on prod | Medium (first deploy always has this) | Smoke test catches immediately; add origin and refresh — no rebuild needed. |
| NCB CORS blocks Better Auth on prod | Medium | Same as above — surface error tells you the origin string to add. |
| Server Action returns 500 on first prod create | Low | Most likely cause is missing env var. Build-time guard catches required NCB vars at startup; AWS/PP vars throw at first call. |
| `(app)/layout.tsx` session check fails on Netlify edge runtime | Low | Layout is `async` Server Component running on Node, not edge; documented in V0.2 design. If Netlify routes it to edge unexpectedly, force Node via `export const runtime = 'nodejs'`. |
| Bootstrap password leaked from chat history | Already happened | Konrad rotates via forgot-password flow on first prod login. Unrelated to V0.3 ship-readiness. |

Rollback for the whole milestone is straightforward: `git revert` the V0.3 commits, `set_rls_policy(workers, 'public_readwrite,shared_readwrite')`, app is back to V0.2 state.

## Forward compatibility

V0.3 prepares us well for:

- **V0.4 hardening:** AWS IAM tightening, rate limits on upload routes, automated tests once the surface stabilises
- **V1.0 multi-table:** RLS-private model is already correct; new tables added with `private` policy from day one
- **Vercel side branch:** the only Vercel-specific work would be `vercel.json` (probably unnecessary) + env vars in Vercel dashboard. Code is host-agnostic.

V0.3 doesn't prepare us for:

- Multi-tenant signup (still single-admin; V1.0)
- Scaling beyond ~200 workers (no real pagination yet)
- Custom domain / branded URL (deferred)
