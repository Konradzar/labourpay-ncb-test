# Design — V0.1 Edit + Delete worker

**Date**: 2026-05-07
**Author**: Konrad (collaborated with Claude)
**Status**: Approved — ready for implementation plan
**Builds on**: `2026-05-07-worker-profile-slice-design.md` (V0)
**First commit on V0 branch this builds on**: `1c91eb4` (Bearer auth fix for `/read/workers/<id>`)

## Purpose

Round out CRUD basics by adding Edit + Delete to the worker profile slice. Per the V0 design doc's "Deferred" table:

> | Edit worker | V0.1 | Add immediately after V0 to round out CRUD basics |
> | Delete worker | V0.1 | Need to think about S3 cleanup behaviour |

This document settles "S3 cleanup behaviour" (answer: leave orphans, consistent with edit-replace policy on line 246 of the V0 design doc) and locks the architectural choices for both flows.

**Explicitly NOT goals**:
- Per-user authorization checks (V0.2 — anonymous mode in V0)
- Custom delete-confirmation modals or type-name confirmation
- S3 garbage collection of orphan files
- Bulk edit/delete on the list page
- Optimistic UI updates
- Undo/restore after delete (would require soft-delete column; current schema has no `deleted_at`)

---

## Decisions log

| # | Decision | Reasoning |
|---|---|---|
| 1 | Edit form lives at separate page `/workers/[id]/edit` (not inline, not modal) | Detail page stays a Server Component; edit form mirrors the proven `AddWorkerForm` pattern. ~95% code reuse. |
| 2 | Delete worker leaves S3 photo + doc as orphans (no cleanup) | Consistent with the existing edit-replace policy ("Orphan-file cleanup probably never, cost is negligible at our scale" — V0 design doc line 246). YAGNI. S3 storage cost at our scale is ~$0.0001/month/worker. |
| 3 | Edit form scope = name + salary + optional file replacement (both files) | Edit-without-files is a half-feature; first photo typo would force delete + re-add (which loses the row's id). Full edit semantics = ~50 new lines of distinct code over `AddWorkerForm`. |
| 4 | Delete confirmation = native browser `confirm()` | Cheapest/simplest. V0 has one user (Konrad) on localhost; cost of a slip is 30s to re-add. TODO note to upgrade to typed-name when V0.3 ships. |
| 5 | Architecture for PATCH/DELETE = Next.js 16 Server Actions (not new API route, not extension of public-data) | Framework-native, no new HTTP route to secure, secret key stays server-side, automatic cache revalidation via `revalidatePath`. Forward-compatible with V0.2 auth (just add `getSessionUser()` check at top of action). |
| 6 | New helper `ncbAuthFetch(pathSuffix, init?)` in `lib/ncb-utils.ts` | Formalises the Bearer-auth pattern from `1c91eb4`. Without it, three files (detail page, update action, delete action) each independently set `Authorization` headers — silent drift over time. |

## Scope summary

**In scope (V0.1):**
- New page `/workers/[id]/edit` with pre-filled form for `name`, `monthly_salary`, `photo`, `id_doc`. Files optional on edit (blank = keep existing).
- Detail page `/workers/[id]` gains "Edit worker" link + "Delete worker" button.
- Server Actions (`app/workers/[id]/actions.ts`) handle PATCH and DELETE against NCB using `Bearer ${NCB_SECRET_KEY}`.
- Delete = DB row only; S3 objects orphaned.
- New manual-test scenarios 9–16 added to `MANUAL_TEST.md`.

## File structure

### New files (4)

| Path | Type | Purpose |
|---|---|---|
| `app/workers/[id]/actions.ts` | Server Action module (`"use server"`) | Exports `updateWorker(id, fields)` and `deleteWorker(id)`. Uses Bearer auth + `revalidatePath` + `redirect`. |
| `app/workers/[id]/edit/page.tsx` | Server Component | Fetches worker by id (same `fetchWorker` pattern as detail page), 404 if missing, renders `<EditWorkerForm worker={…} />`. |
| `app/workers/[id]/edit/EditWorkerForm.tsx` | Client Component | Pre-filled form, file inputs optional, calls `/api/upload-url` for any new files, then invokes `updateWorker` Server Action. |
| `app/workers/[id]/DeleteWorkerButton.tsx` | Client Component | Tiny — wraps a form whose action is `deleteWorker.bind(null, id)`. Button has `onClick` that gates on `confirm()`. |

### Modified files (4)

| Path | Change |
|---|---|
| `app/workers/[id]/page.tsx` | Add "Edit worker" link near top, `<DeleteWorkerButton id={…} name={…} />` near bottom. Existing structure untouched. |
| `lib/ncb-utils.ts` | Add `ncbAuthFetch(pathSuffix, init?)` helper. Refactor detail page's `fetchWorker` to use it (DRY). |
| `lib/types.ts` | Add `WorkerUpdateInput` type. |
| `MANUAL_TEST.md` | Append scenarios 9–16. |

**Why `actions.ts` lives at `app/workers/[id]/actions.ts` (not `.../edit/actions.ts`)**: delete is invoked from the detail page (parent), edit from the edit page (child). Both share these actions, so the action file lives at the lowest common ancestor.

**Why no changes to `app/api/public-data/[...path]/route.ts`**: Server Actions bypass that route entirely. The route stays anonymous-only as designed; we add a privileged path via Server Actions instead of widening the existing public surface.

## Data flow — Edit worker

```
1. User on /workers/4 clicks "Edit"
   ↓
2. /workers/4/edit (Server Component)
   - fetchWorker(4) via ncbAuthFetch (Bearer auth)
   - 404 if missing
   - Render <EditWorkerForm worker={…} />
   ↓
3. EditWorkerForm (Client Component) pre-fills inputs
   User changes some fields, optionally picks new files
   User clicks "Save"
   ↓
4. onSubmit handler in EditWorkerForm
   a. Validate (name not empty, salary > 0)
   b. IF new photo file selected:
      i.   POST /api/upload-url → { url, key }
      ii.  PUT file → S3 (browser → S3 directly)
      iii. photo_key = new key
      ELSE photo_key = worker.photo_key (unchanged)
   c. Same flow for id_doc
   d. Call updateWorker(worker.id, { name, monthly_salary, photo_key, id_doc_key })
   ↓
5. updateWorker Server Action (server-side)
   - PATCH ${dataApiUrl}/update/workers/${id}
       Authorization: Bearer ${NCB_SECRET_KEY}
       body: JSON of changed fields
   - revalidatePath('/')
   - revalidatePath(`/workers/${id}`)
   - redirect(`/workers/${id}`)
   ↓
6. Browser lands on /workers/4 with fresh data
```

**Key invariants:**
- Files upload BEFORE the action runs (skip Next.js server during file transit; S3 PUT is browser → S3 direct via presigned URL).
- The Server Action receives only field values, never File objects.
- `worker.id` is captured at form mount and passed as the first action argument — no URL parsing inside the action.

## Data flow — Delete worker

```
1. User on /workers/4 clicks "Delete worker" (DeleteWorkerButton)
   ↓
2. confirm("Delete worker Piet Pol? This cannot be undone.")
       Cancel → e.preventDefault(), nothing happens
       OK     → form submits
   ↓
3. <form action={deleteWorker.bind(null, id)}> POSTs
   Next.js dispatches Server Action
   ↓
4. deleteWorker(id) runs server-side
   - DELETE ${dataApiUrl}/delete/workers/${id}
       Authorization: Bearer ${NCB_SECRET_KEY}
   - On 200: revalidatePath('/'); redirect('/')
   - On non-200: throw → error UI / boundary
   ↓
5. Browser lands on / with the worker absent from the list
```

**S3 cleanup**: none. `worker.photo_key` and `worker.id_doc_key` remain in S3 as orphans. Per Decision 2.

## Error handling

| Scenario | Where it happens | User experience | Recoverable? |
|---|---|---|---|
| **A.** User opens `/workers/999/edit` (worker doesn't exist) | Server-side fetch returns 404 | Next.js `not-found.tsx` → 404 page | Yes — back to `/` |
| **B.** Edit form: name empty or salary ≤ 0 | Client validation | Inline red text under the bad field, submit aborts before network | Yes — fix and resubmit |
| **C.** Edit form: file too large or wrong type | Client validation (mirrors AddWorkerForm) | Inline red text "File must be JPG/PNG/PDF, max 5 MB" | Yes — pick another file |
| **D.** Edit: presigned URL request to `/api/upload-url` fails | Client fetch | Inline error "Couldn't prepare upload — please try again." Save button re-enabled | Yes — retry |
| **E.** Edit: S3 PUT fails | Client fetch (presigned URL) | Inline error "Upload failed — check your connection and try again." Save re-enabled. Earlier-success files become orphans (acceptable) | Yes — retry |
| **F.** Edit: Server Action's PATCH returns non-2xx | `updateWorker` throws | Next.js error overlay (dev) / `error.tsx` boundary (prod) | Yes — retry. New S3 files become orphans |
| **G.** Delete: NCB DELETE returns non-2xx | `deleteWorker` throws | Same as F | Yes — retry |
| **H.** Delete: worker already deleted (race / refresh) | NCB returns 404 | Action throws "NCB delete worker failed: 404 …" | Idempotent — refresh `/`. Could special-case 404 → success in the action; YAGNI for V0.1 |
| **I.** Either action: NCB unreachable (network) | `fetch` rejects | Action throws → error boundary | Yes — retry |

### Conventions

- **Field-level errors** (B, C, D, E) appear inline under the field, in `#d63031`, font size 0.85rem. No alert boxes.
- **Server Action errors** (F, G, H, I) bubble to Next.js's error boundary. We do NOT catch them in the action — letting them surface gives the user the actual NCB response, which is invaluable for debugging.
- **No "unknown error" wrappers.** If NCB returns `500 Internal Server Error: <html>…`, that's exactly what shows. Ugly but honest.
- All Server Action errors include the NCB response body in the thrown message: `throw new Error(`NCB ... failed: ${res.status} ${await res.text()}`);`
- No new logging library. Next.js dev server logs (`.next/dev/logs/next-development.log`) are sufficient.

### Edge cases knowingly NOT handled in V0.1

- **Concurrent edit** (last-write-wins) — NCB has no `version` / `updated_at` column on `workers`; optimistic concurrency requires schema change. Acceptable: only one user until V0.2.
- **Partial file replacement on retry** — if S3 PUT succeeds and DB update fails and user retries, second attempt re-uploads to a *new* key. Two orphans. Acceptable per orphan policy.
- **Cancel mid-upload** — upload continues in browser background. Files orphan. Acceptable.

```sql
SELECT id, name, monthly_salary, photo_key, id_doc_key FROM workers WHERE id = <X>;
```

SQL is ground truth — visual inspection can mislead (today's photo-vs-doc-keys false alarm proved this).

### Explicitly NOT tested in V0.1

- Concurrent edits (two tabs editing same worker)
- Browser autofill quirks
- Mobile responsive layout (V0 is desktop-only)
- Accessibility (deferred until V0.3 deploy)
- Performance at scale (revisit at 100+ workers)

## Forward-compatibility notes

What this design **prepares us well for** in V0.2 / V0.3:

- **V0.2 per-user auth flip**: each Server Action gets a one-line `const user = await getSessionUser(...); if (!user) throw new Error("Unauthorized");` at the top. The action's location (vs an API route) doesn't change the work meaningfully.
- **V0.3 deploy hardening**: Server Actions are server-side by definition; secret keys stay out of the browser bundle. No additional refactor needed for the "flip RLS to private" gate.
- **V1.0 more tables**: same pattern (`app/<entity>/[id]/actions.ts`) scales linearly.

What this design does **NOT** prepare us for (and shouldn't, per YAGNI):

- Real-time updates (multiple users editing simultaneously) → V2+
- Soft delete with restore → V2+
- Audit log of who-changed-what-when → V2+ (would need `updated_at`, `updated_by`, history table)
